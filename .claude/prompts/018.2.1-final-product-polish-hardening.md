# Finding 1 — Retry must not depend on URL rotation

## Problem

The current image-error Retry only calls `refetch()`. The image is remounted by an effect only when `freshPhoto.originalUrl` changes. A presigner may return the exact same URL for identical inputs requested within the same signing-time interval; in that case the effect does not run, `imgRevision` does not change, `imgState` remains `error`, and Retry is a dead button.

## Required behavior

1. Treat completion of an explicit successful Retry as the remount signal, not string inequality of `originalUrl`.
2. On Retry:
   - capture the current photo ID;
   - enter a bounded retry/loading state and prevent duplicate concurrent Retry clicks;
   - refetch authenticated `GET /photos/{id}`;
   - wait for that request to succeed;
   - if the viewer is still on the captured ID, increment the image revision and set image state to loading even when the returned `originalUrl` is byte-identical;
   - render the current query result only after the refresh completes.
3. Never append cache-busting parameters to a presigned URL. The React element key/revision, not URL mutation, forces the browser retry.
4. If explicit refetch fails, keep a recoverable error state and allow another Retry. Do not silently reuse the failed image or leak API/presigned URL details.
5. Disable Retry while its refetch is in flight. Do not introduce automatic refetch loops.
6. Ignore late Retry completion after navigation, close/unmount, or a change to another photo ID. It must not reset or remount the new photo.
7. Preserve initial `refetchOnMountOrArgChange`, `currentData` stale-argument protection, previous/next navigation, Escape/close, focus restoration, bbox alignment, and signed-URL secrecy.
8. Simplify the old URL-change effect/ref bookkeeping if it is no longer necessary. Keep one understandable retry lifecycle rather than two competing remount mechanisms.

# Finding 2 — no false empty prediction state while loading

## Problem

While `freshPhoto` is undefined, the sidebar calls `PredictionList` with `[]`, so it displays `No detections` even though predictions are still unknown.

## Required behavior

1. Render prediction data only when a matching `freshPhoto` exists.
2. While the initial photo fetch or explicit Retry is pending, show a neutral accessible loading state in the sidebar (or leave the details region empty with an appropriate busy label). Do not show `No detections` until a successful response actually contains an empty predictions array.
3. On API failure, do not show `No detections`; the existing recoverable media error is sufficient, with the sidebar remaining neutral.
4. After success, preserve the real prediction list, confidence values, class highlighting, and genuine `No detections` behavior for an empty returned array.

# Finding 3 — normalize clamped percentage text

## Problem

`applyConfidence` clamps the ratio but updates the input text only indirectly through a Redux change. If Redux already contains `1` and the user enters `150`, the computed ratio is still `1`, no dispatch occurs, and the field incorrectly remains `150` while the active filter is `100%`. The equivalent issue exists for an existing `0` and a negative input.

## Required behavior

1. After strict numeric parsing, always write the clamped percentage back to `confidenceInput`, regardless of whether a Redux dispatch is needed.
2. Examples:
   - current Redux `1`, input `150` → visible `100`, Redux remains `1`, no duplicate dispatch;
   - current Redux `0`, input `-10` → visible `0`, Redux remains `0`, no duplicate dispatch;
   - current Redux `0.7`, input `70` → visible `70`, no duplicate dispatch.
3. Preserve the percentage-to-ratio boundary, empty-input clearing, `Number` parsing, Enter/blur/Apply behavior, reset, active summary, and `0–100` accessibility attributes.
4. Do not change Redux/API/OpenAPI confidence representation from `[0,1]`.
