# Goal

Harden the SQLite repository from task 004 by fixing four review findings: safe absolute file-URI construction, a real read-only assertion, exact cursor-boundary preservation, and deterministic prediction ordering.

Task 004 tests currently pass, but review found these gaps:

1. `Open` interpolates the caller path directly into `file:%s?...`; it does not make the path absolute or safely encode URI-significant characters such as spaces, `?`, or `#`.
2. `TestOpen_readOnly` never attempts a write through the repository connection, so it does not prove that `mode=ro` is effective.
3. Cursor creation converts the database's original `captured_at` text through `time.Time` and formats it as UTC. The keyset query compares the cursor back to the raw SQLite text column, so textual normalization could change the boundary and cause omissions or duplicates for valid RFC 3339 values with offsets or formatting differences.
4. Predictions are ordered by `rowid`, although their stable primary key `id` is available. The repository contract should not depend on SQLite's internal rowid.

Before editing, briefly restate the four findings, intended fixes, acceptance criteria, and expected changed files.

# Requirements

## Safe read-only DSN

1. In `Open`, convert the supplied database path to a cleaned absolute path before opening it.
2. Construct the SQLite file URI with `net/url` or another standard-library URI-safe approach. Do not interpolate a raw path into a DSN with `fmt.Sprintf`.
3. Preserve `mode=ro` explicitly in the URI query.
4. Use only modernc.org/sqlite-supported query parameters. Do not rely on an unverified `_foreign_keys=1` parameter spelling.
5. Foreign-key enforcement is not required for read-only SELECT operations. If it is retained, configure it using the driver's documented pragma mechanism.
6. Return descriptive wrapped errors for absolute-path resolution, open, and ping failures without exposing them as public application errors.
7. Add tests proving that paths containing spaces and URI-significant characters such as `#` and `?` open successfully.
8. Keep the repository compatible with ordinary relative and absolute Linux paths.

## Real read-only verification

9. Replace the indirect `TestOpen_readOnly` assertion with a real write attempt through the repository's own unexported `db` handle from the same-package test.
10. Attempt a harmless INSERT into the temporary fixture and assert that SQLite rejects it because the database is read-only.
11. Verify after the failed INSERT that the row was not created and normal repository reads still work.
12. Never perform this write attempt against `dataset/predictions.db`; use a temporary fixture only.

## Exact cursor boundary

13. Preserve the exact `captured_at` string scanned from SQLite for cursor generation and comparison.
14. Cursor decoding must validate that the stored text is a valid RFC 3339/RFC 3339 Nano timestamp, but it must return or otherwise retain the original validated string for the SQL boundary.
15. Do not normalize cursor timestamps to UTC before using them in the keyset WHERE clause.
16. Continue mapping the same value to `domain.Photo.CapturedAt` as `time.Time` for domain consumers.
17. Keep raw database timestamp text internal to the repository; do not add it to `domain.Photo`.
18. Use a small private row/boundary structure if necessary to carry both the domain photo and original timestamp through `queryPhotos`.
19. Build `NextToken` from the exact raw timestamp belonging to the last returned photo.
20. Keep ordering and comparison consistent with SQLite:

```sql
ORDER BY captured_at DESC, id DESC
```

and the existing keyset boundary.
21. Add a pagination test using valid RFC 3339 timestamp text that is not already in canonical UTC `Z` form. Traverse all pages and assert no duplicates or omissions.

## Strict cursor JSON decoding

22. After decoding the first JSON value, perform a second decode and require `io.EOF`.
23. Reject trailing non-whitespace text and a second valid JSON value.
24. Continue allowing harmless trailing JSON whitespace.
25. Keep `DisallowUnknownFields`, version checking, required fields, timestamp validation, and UUID validation.
26. Add explicit tests for:
    - a second JSON object;
    - trailing non-whitespace text;
    - trailing whitespace accepted.

## Deterministic prediction ordering

27. Select the prediction primary key `id` as needed and order batched predictions by:

```sql
photo_id, id
```

28. Do not expose prediction IDs in `domain.Prediction` or API data.
29. Add a test that inserts predictions in an order different from their IDs and verifies returned predictions follow deterministic ID order.

## Preservation

30. Preserve all task-004 behavior:
    - read-only repository lifecycle;
    - typed validation/not-found/internal errors;
    - same-prediction `EXISTS` filters;
    - inclusive confidence filter;
    - limit plus one pagination;
    - all predictions returned for matched photos;
    - batched prediction loading;
    - non-nil empty prediction slices.
31. Do not weaken or delete existing meaningful tests.

# Acceptance criteria

- Relative and absolute database paths are converted into safe read-only SQLite file URIs.
- Fixture paths containing spaces, `#`, and `?` work.
- A write through the repository's connection fails with a read-only SQLite error and changes no data.
- Cursor pagination reuses the exact validated timestamp text scanned from SQLite.
- Pagination with a non-UTC-offset RFC 3339 timestamp has no duplicates or omissions.
- Cursor parsing rejects any second JSON value or trailing junk while allowing whitespace.
- Prediction ordering uses the declared primary key rather than `rowid`.
- All existing and new tests pass under the race detector.
- The supplied dataset remains unchanged.
- No executable or other build artifact remains in `backend/`.
