1. Do not use the possibly stale `originalUrl` from the current list page as the viewer image source.
2. On opening a photo and whenever viewer navigation selects another photo, obtain that photo through the existing authenticated `GET /photos/{photoId}` RTK Query endpoint. Wait for the current ID's fresh response before loading the original.
3. Never display a previous photo's query result while the current photo request is pending. Guard against fast previous/next navigation and out-of-order responses by associating data with the current canonical photo ID.
4. Preserve the current dialog behavior: contained-image bbox geometry, prediction list, previous/next bounds, Escape/close, keyboard navigation, focus restoration, loading state, and malformed-data isolation.
5. A Data API failure must produce a recoverable viewer error state. Retry must refetch `GET /photos/{id}` rather than repeatedly loading the same stale URL.

7. Avoid request loops: one fetch on open/ID change, and additional fetches only after an explicit retry or normal RTK Query cache/refetch policy.
8. Continue using the list-page photo order for previous/next navigation. Cross-page viewer navigation is out of scope.

# Requirement 2 — percentage confidence input

Implement one clear representation boundary:

- Human-facing input and summaries: percent in `0–100`.
- Redux state, RTK Query arguments, URL query, and OpenAPI contract: ratio in `0–1`.

Required behavior:

1. Label the field clearly as `Minimum confidence (%)` (or equally explicit wording), use a numeric input with visible `%` affordance, `min=0`, `max=100`, and `step=1`.
2. Use an empty field to mean no confidence filter. A user-entered `70` must commit exactly `0.7` to Redux and produce `minConfidence=0.7` in the API request.
3. When Redux is externally set to `0.7`, display `70` in the field. Reset must return the field to empty.
4. Parse strictly with `Number`, reject non-finite/invalid text, and clamp committed values to `0–100` before dividing by 100. Do not use permissive `parseFloat` behavior.
5. Preserve the existing atomic blur/Apply behavior: blur followed by Apply must not dispatch the same value twice. Enter and Apply must use the same conversion path.
6. Keep the active-filter summary in percent form (for example `≥70%`). Do not change backend filter semantics: class and confidence still have to match the same prediction, while responses retain all predictions.
7. Do not add a second ratio input, range slider, preset chips, dependency, or new global state. A single percentage field is the simplest accessible UI for this scope.
