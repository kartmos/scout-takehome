# Required schematic

Render a simple semantic HTML/CSS layout, not a geographic SVG/canvas:

```text
┌──────────────────────── Bed 1 ────────────────────────┐
       ●1       ●2       ●3       ●4       ●5
┌──────────────────────── Bed 2 ────────────────────────┐
       ●6       ●7       ●8       ●9       ●10
┌──────────────────────── Bed 3 ────────────────────────┐
```

1. Render exactly three long horizontal rectangular beds in this order: `Bed 1`,
   `Bed 2`, `Bed 3`. Each has a visible border, restrained greenhouse/olive fill,
   and centred label. Do not fabricate plants, aisles, dimensions, coordinates, or
   a realistic floor plan.
2. Render exactly ten native `<button>` point controls:
   - points 1–5 in one row between beds 1 and 2;
   - points 6–10 in one row between beds 2 and 3.
3. Each point visually consists of:
   - a larger, lighter, semi-transparent outer circle;
   - a smaller opaque inner circle;
   - its number clearly visible in or immediately beside the control;
   - a compact assigned-photo count that does not obscure the hit target.
4. Use the application accent color rather than a pest-class color: a point may
   contain photos of several classes, so class coloring would be misleading.
5. Hover behavior should feel like the existing filter buttons: stronger border/
   color/background contrast with a short restrained transition. Keyboard
   `:focus-visible` gets an equally clear outline. Do not rely on hover alone.
6. The selected point gets a persistent high-contrast ring/state distinct from
   hover. Empty points remain visible; they may be disabled only when their count
   is zero and must expose an accessible explanation.
7. Do not reintroduce 40×40 axes, `x/y/h`, clusters, markers derived from API
   coordinates, tooltips with coordinates, zoom, pan, or reset-view controls.

# Deterministic pseudo-random photo assignment

The UI should appear randomly distributed, but assignment must never change across
renders, filter changes, drawer mode changes, or page reloads.

1. Implement one pure deterministic helper based only on canonical `photo.id`, for
   example a small stable string hash followed by:

   ```text
   pointNumber = positiveHash(photo.id) % 10 + 1
   ```

2. Do not use `Math.random`, array index, current list order, timestamp, active
   filters, or React state to assign a point.
3. The same photo ID must always map to exactly one integer point in `1..10`.
4. The helper must handle unexpected non-canonical strings deterministically
   without throwing, while normal production data remains canonical.
5. Assignment exists only in frontend memory/code. Do not write it to Redux,
   localStorage, URL, database, API, photo objects, or generated types.

# Bounded map dataset

The normal gallery remains cursor-paginated at its current page size. The schematic
map may use one separate bounded first-page query so the current 50-photo demo is
distributed across all ten points without changing the database.

1. Use the existing `useListPhotosQuery` endpoint with:
   - `limit: 200`;
   - current `classId` when selected;
   - current `minConfidence` when selected;
   - no cursor traversal.
2. Start this bounded map query only after the map has first been opened, or while
   a point filter is active. A never-opened hidden map must not eagerly request 200
   photos.
3. Never follow `next_token`. If it exists, disclose in the map/footer that point
   counts/results cover the first 200 matching photos, not the whole catalog.
4. Current project data has 50 photos, so the bounded query covers the demo dataset.
   Do not hardcode the number 50 or any photo IDs.
5. Existing class and confidence filters are included in this query, so point counts
   update to the currently matching photo set.
6. Use `currentData`, not stale prior-argument `data`, so point counts and selected
   results never flash results from old filters.
7. Provide compact loading/error states inside the drawer. A map-query failure must
   not break the normal paginated gallery.

# Point selection as a local gallery filter

1. Store `selectedPoint: 1..10 | null` in `GalleryPage` local state. Do not add it
   to the Redux filter slice.
2. When no point is selected, the normal gallery and cursor pagination work exactly
   as before.
3. When a point is selected, derive:

   ```text
   pointPhotos = boundedMapQuery.items.filter(
     photo => assignedPoint(photo.id) === selectedPoint
   )
   ```

   Render these photos through the existing `PhotoCard` with the current
   `matchingClassId` and `minConfidence` props.
4. The effective result is therefore the intersection of:
   - deterministic map point;
   - existing class filter;
   - existing minimum-confidence filter.
5. While a point is selected, hide/replace normal cursor pagination because the
   displayed result comes from the bounded map query, not the current 24-photo
   page. Do not mutate or lose the saved normal page/cursor state; clearing the
   point returns the user to the same normal gallery page.
6. Above the filtered gallery show an explicit location-filter indicator, for
   example:

   ```text
   [ Map point 4 · 6 photos ]  [ × Clear point ]
   ```

   The location filter is internal/client-only but must not be visually invisible;
   otherwise hiding the map would make the reduced gallery inexplicable.
7. `Clear point` is a real non-underlined button. It clears only location selection.
   Existing `Clear filters` continues clearing class/confidence. Neither action
   silently performs the other.
8. If class/confidence changes make the selected point empty, keep the selected
   point and render a clear empty state with both `Clear point` and the existing
   relevant filter-clearing action available.
9. Point counts always reflect the same current bounded map query used by the
   selected-point gallery.
10. The selected point remains highlighted even if the drawer is hidden and returns
    highlighted when reopened.

## Expanded

1. Expand to a modal/dialog approximately the same maximum dimensions as

2. Use a native `<dialog>` or an equivalently accessible modal with backdrop,
   Escape close, focus containment, and focus return to the expand/launcher control.
3. Header controls:
   - `Return to mini-map`;
   - `Hide map` or close.
4. Render the same schematic component, point counts, selection behavior, and
   states as compact mode. Only layout/scale changes; do not duplicate business
   logic or create separate assignments.
5. Switching compact ↔ expanded must preserve selected point, loaded map data,
   query cache, and current filters without remount-driven reassignment.

## Responsive behavior

1. On narrow/mobile screens, compact mode becomes a bottom sheet spanning the
   available width rather than overflowing horizontally.
2. Preserve two rows of five points if practical; at very narrow widths reduce
   gaps and point size without changing numbering/order or creating horizontal page
   scroll.
3. Expanded mode remains usable in landscape and portrait with internal scrolling
   only when necessary.

# Accessibility

1. All ten points are native buttons with accessible names including point number
   and current photo count, for example `Map point 4, 6 photos`.
2. Selected state uses `aria-pressed` or another correct native pattern.
3. Bed rectangles are descriptive, not interactive; do not put them in tab order.
4. Map launcher, expand, return-to-mini, hide, retry, point, and clear-point controls

5. Do not create a keyboard trap in compact mode. Expanded modal follows normal
   dialog focus/Escape/return rules.
6. Counts should not create noisy live-region announcements on every intermediate
