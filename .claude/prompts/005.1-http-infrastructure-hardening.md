# Goal

Harden the HTTP infrastructure from task 005 by correcting panic diagnostics and their regression tests: log the recovered panic exactly once as a diagnostic failure, keep it out of the client response, avoid a duplicate internal-error diagnostic, and preserve the required access-completion log.

Task 005 passes its current tests, race detector, and vet, but review found that the tests encode behavior opposite to the task contract:

1. `panicRecoveryMiddleware` logs `"panic recovered"` with the request ID but omits the recovered panic value. Task 005 requirement 38 requires the panic to be present in the server-side error diagnostic while never appearing in the HTTP response.
2. Before headers are committed, recovery first emits `"panic recovered"` and then calls `WriteError` with `apperror.NewInternal(nil)`. `WriteError` emits another `"internal error"` diagnostic, so one panic creates two diagnostic failure records before the separate access-completion event.
3. `TestPanicRecovery_BeforeWrite` explicitly asserts that the panic value must not appear in logs, contradicting task 005. It also does not assert the diagnostic count, so the duplicate is not detected.

The access log entry `"request completed"` is a separate required event and is not the duplicate to remove. A recovered 500 should produce:

- exactly one diagnostic failure record containing the request ID and panic value;
- exactly one access-completion record containing the final 500 status;
- one safe OpenAPI-compatible 500 response containing neither the panic value nor any internal detail.

# Requirements

## Panic before response commitment

1. Continue recovering every downstream panic except `http.ErrAbortHandler`.
2. Emit exactly one diagnostic failure log for the recovered panic.
3. The diagnostic must be error level and include structured fields for:
   - the request ID;
   - the recovered panic value.
4. Do not log request headers, query strings, bodies, API keys, cookies, authorization values, or unrelated request data.
5. Write the response through the centralized OpenAPI-compatible error path or a minimal internal variant of that path.
6. Return status `500`, code `InternalServerError`, the safe public message, and the same request ID as the `X-Request-ID` header.
7. Never include the panic value, its type formatting, a stack trace, or any internal cause in the response body or headers.
8. Do not produce a second diagnostic such as another `"internal error"` record for the same recovered panic.
9. Preserve the required access log. The outer access logger must still emit one `"request completed"` event at error level with status 500, response bytes, route, method, duration, and request ID.

One acceptable small design is to pass an internal error retaining a safe textual representation of the panic into `WriteError` and let `WriteError` be the sole diagnostic logger before the access-completion event. Another design is acceptable only if it has the same one-diagnostic behavior and does not weaken centralized response shaping.

## Panic after response commitment

10. When headers or body have already been written, do not append an error JSON document and do not change the committed status.
11. Still emit exactly one diagnostic failure log containing request ID and panic value.
12. Return safely after recovery without logging a second internal-error diagnostic.
13. Preserve the partial-response byte/status accounting in the access-completion log.

## `http.ErrAbortHandler`

14. Continue re-panicking `http.ErrAbortHandler` unchanged so `net/http` can close the connection without its normal panic logging.
15. Do not convert it into a JSON 500 response.
16. Do not add a recovery diagnostic before re-panicking it.

## Regression tests

17. Correct `TestPanicRecovery_BeforeWrite`: the panic value must be absent from the response but present in the diagnostic log.
18. Assert that the request ID is present in the diagnostic log.
19. Parse structured logs or use a purpose-built in-memory `slog.Handler` where practical; avoid fragile assertions that cannot distinguish separate records.
20. Assert exactly one diagnostic failure record for a panic before write.
21. Assert exactly one `"request completed"` access record and verify its status is 500.
22. Update the after-write test to assert:
   - the panic value and request ID are in exactly one diagnostic failure record;
   - no JSON error is appended to the partial body;
   - the access-completion event still reflects the committed status and bytes.
23. Preserve the `http.ErrAbortHandler` re-panic test and assert that no recovery diagnostic is emitted for it.
24. Add a regression assertion that a distinctive panic string never appears in the response body or response headers.
25. Do not weaken existing tests for request IDs, authentication, CORS, error mapping, secret redaction, response-controller unwrapping, or access-log metadata.

# Acceptance criteria

- A panic before write returns the same safe typed 500 response required by task 005.
- The panic value is visible only in one server-side diagnostic record and never in the HTTP response.
- The diagnostic record contains the request ID.
- No duplicate internal-error diagnostic is emitted for the same panic.
- The separate access-completion event remains present and records status 500.
- A panic after write does not append JSON and produces one diagnostic plus one access-completion event.
- `http.ErrAbortHandler` is re-panicked without a JSON response or recovery diagnostic.
- Existing HTTP/config behavior remains passing under repeated and race-enabled tests.
- Only the allowed HTTP middleware/test files change.
- No dependency or build artifact is introduced.
